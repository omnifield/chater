# Ревью — Шаг 4 (chater websocket). Вердикт: ПРИНЯТ (PR #6)

**От:** workspace-архитектор + user.

## Проверено под `-race` (интеграционные тесты + прочтение кода)
- ✅ `TestWSLiveDeliveryToTwoSubscribers` — два подписчика получают live-сообщение (ядро).
- ✅ Не-участник → **403 до апгрейда**; без auth → **401**; дисконнект чистит подписку (count→0).
- ✅ Хаб: fan-out, изоляция комнат, дроп при полном буфере — юнит-тесты. **Гонок нет** (весь пакет `-race` зелёный).
- ✅ Код: bounded-буфер 32 + неблокирующий `publish` (не стопорит бродкаст/HTTP), RWMutex + очистка пустых
  комнат, `requireParticipant` до апгрейда, `CloseRead`-ctx на дисконнект, ping-keepalive, конверт
  `{type:"message"}` единым shape с HTTP-DTO. Канал не закрывается (нет паники).

Блокеров нет.

## Ответы на 3 решения
1. **Дроп кадров для медленного клиента (буфер 32), HTTP-история как источник правды — верно.** Live-feed
   best-effort, полнота/порядок добираются историей. Стандартный и правильный выбор для v0.
2. **WS-auth Bearer-заголовком; браузер не может — задокументировано; агенты/Postman ок.** ✓ Скоуп верный;
   браузерный путь (subprotocol/query-token) — с реальной identity, не сейчас.
3. **Хаб single-instance; мульти-инстанс pub/sub за скоупом.** ✓ Для v0 (один процесс) правильно.
