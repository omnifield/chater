# chater — Баг фронта: кнопка Create не шлёт запрос + приложение залипает на ошибке загрузки

Приоритет: блокер (user не может пользоваться UI). Ветка от свежего main, PR, STOP + отчёт с
НАЙДЕННОЙ первопричиной. Ревью — workspace-архитектор + user.

## Симптом (user подтвердил дважды)
- При клике **Create** в браузере **НЕ уходит НИ ОДНОГО сетевого запроса** (пусто в Network-таб).
- UI при этом пишет **«Cannot reach the server.»**, вверху — **«Could not load rooms: Cannot reach the server.»**
- Перезапуск vite не помогает; состояние не восстанавливается.

## Что НЕ является причиной (проверено сервер-сторона workspace-архитектором)
- backend :8020 живой (`/chater/healthz` 200).
- vite-proxy :5173 отдаёт `/chater/*` → **200**, в т.ч. с чужим Host (`x.devtunnels.ms`) после фикса
  `allowedHosts` (PR #8, в main). Т.е. **бэкенд и прокси НЕ виноваты — баг клиентский.**

## Задача
Открой фронт в браузере сам (`go run ./cmd/chater` :8020 + `pnpm -C web dev --host`), **воспроизведи**
и найди **первопричину**, почему:
1. клик Create не порождает fetch (запроса нет вообще), и
2. приложение залипает в «Cannot reach the server» без восстановления.

Кандидаты (проверь, не гадая — воспроизведи и подтверди):
- `createResource(listRooms)` **латчит терминальную ошибку** при первом фейле (например, бэкенд ещё
  компилировался/не слушал на момент загрузки) и **никогда не ретраит** → список скрыт за error-fallback,
  а юзер застрял. Нужен retry/refetch и устойчивость к транзиентному первому фейлу.
- не глотается ли исключение в рендере/хендлере так, что интерактивность/обработчик submit мертвы
  (тогда клик реально не даёт fetch).
- проводка `<form onSubmit={create}>` / `type="submit"` в реальном рендере при error-state.

## DoD
- **Воспроизведён** «no request on Create» локально, назван корень (в отчёте — что именно и почему).
- Фикс: Create всегда шлёт запрос и создаёт комнату; приложение **восстанавливается** после транзиентной
  ошибки первой загрузки (кнопка Retry / автоперефетч), без «вечного» залипания.
- Тест на путь ошибка-загрузки → восстановление (и что Create вызывает клиент).
- `go`/`web` CI зелёные. Ветка + PR. STOP → отчёт с первопричиной.

## Не в скоупе
Инфра/туннель/порты/gateway — НЕ трогать. Это чисто клиентский баг.
